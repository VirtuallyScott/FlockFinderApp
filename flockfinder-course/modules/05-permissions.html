Permission Granted

Why iOS makes your app beg for access — how the permission system works, and what happens when it says no.

iOS Has a Bouncer at Every Door

Imagine every sensitive hardware feature on your iPhone — GPS, microphone, camera, Bluetooth, motion sensors — has its own bouncer at the door. No app gets in without the user explicitly approving it. Even after approval, the user can revoke access at any time from Settings.

FlockFinder needs three types of hardware access to work. Each one requires the user's explicit consent:

📶

Bluetooth

Required to scan for and connect to the ESP32 hardware. Without this, the app literally cannot receive any detections.

📍

Location

Required to tag detections with GPS coordinates. The app requests "When In Use" — only while the app is open.

🏃

Motion & Activity

Optional but useful — lets the app record whether you were driving, walking, or stationary when each camera was detected.

Telling Apple Why You Need Access

iOS requires every app to explain why it needs each permission. These explanations live in a configuration file called Info.plist. Apple reviews them — and if they seem suspicious or vague, your app gets rejected from the App Store.

NSBluetoothAlwaysUsageDescription "FlockFinder needs Bluetooth to connect to your FlockFinder device." — Required for any BLE use on newer iOS.
NSLocationWhenInUseUsageDescription "FlockFinder needs your location to tag detected surveillance devices." — Shown when requesting foreground location.
NSLocationAlwaysAndWhenInUseUsageDescription For the optional background mode — "FlockFinder needs location access to log detections while you drive." Background requires extra justification.
NSMotionUsageDescription "FlockFinder uses motion to record your direction and speed during detection." — For CoreMotion access.
⚠️
The Missing Plist Key Crash

If an app calls a permission API without the matching Info.plist key, iOS immediately crashes the app with a hard error. BLEManager.swift even has a debug check: if Bundle.main.object(forInfoDictionaryKey: "NSLocationWhenInUseUsageDescription") == nil — it prints a warning before crashing so you know exactly what's missing. If you see cryptic permission crashes, Info.plist is always the first place to check.

The Permission Request in Code

Here's how LocationManager asks for permission — and what happens at each state.

CODE 

func startTracking() {
    guard authorizationStatus == .authorizedWhenInUse ||
          authorizationStatus == .authorizedAlways else {
        requestAuthorization()
        return
    }
    locationManager.startUpdatingLocation()
    locationManager.startUpdatingHeading()
}
PLAIN ENGLISH

Function to start GPS tracking — called when the app opens.

Safety check: if the user has NOT granted location permission...

...(checking for either "when in use" or "always" access)...

...then ask for permission first (this shows the iOS popup)...

...and stop here — don't try to use GPS without permission.

End of the safety check block.

Permission confirmed — start receiving GPS position updates.

Also start receiving compass heading updates.

End of function.

Codebase Quick Reference

The master map — every file, what it owns, and where to find any feature or behavior in the FlockFinder codebase.

FlockFinder/ — The iOS app source
FlockFinderApp.swift— App entry point. Creates all managers. Wires them into the view tree via .environmentObject()
Info.plist— Privacy permission strings (Bluetooth, Location, Motion). Required by Apple before any hardware access.
Managers/ — Hardware access. No UI.
BLEManager.swift— CoreBluetooth. Scans, connects, receives JSON detections from ESP32. Owns recentDetections[].
LocationManager.swift— CoreLocation. GPS position, speed, heading. Updates every 5 meters.
DatabaseManager.swift— SQLite3. Reads/writes flockfinder.sqlite in app Documents folder. Singleton.
MotionManager.swift— CoreMotion. ⚠️ DEPRECATED. GPS replaced its use. Kept for reference only.
Views/ — SwiftUI screens + orchestration
ContentView.swift— Root TabView. 4 tabs: Scanner | Map | History | Settings.
DetectionCoordinator.swift— Orchestrates each detection: GPS lookup → confidence check → DB save → audio → haptics.
MapView.swift— MapKit. Plots all saved detections as map pins. North-up and heading-up modes.
HistoryView.swift— Searchable list of all detections. Filter by device type. Export to CSV / JSON / GPX.
SettingsView.swift— Preference UI. Reads/writes AppSettings singleton.
AppSettings.swift— NSUbiquitousKeyValueStore. iCloud-synced preferences (confidence threshold, audio, map mode). Singleton.
AudioAlertManager.swift— AVFoundation + UIKit. Detection sounds and haptic feedback. Works with CarPlay and Bluetooth audio.
iCloudManager.swift— CloudKit. Copies flockfinder.sqlite to iCloud Drive. Auto-backup throttled to once per hour.
DebugStreamView.swift— Live raw BLE stream: every Wi-Fi device seen, every BLE device, every channel hop. Developer tool.
Models/ — Data shapes
FlockDetection.swift— The FlockDetection struct (18 fields). The DeviceType enum (12 camera brands + Unknown).
📁
Where Data Lives on Disk

The SQLite database: Documents/flockfinder.sqlite inside the app's sandbox. iCloudManager copies it to iCloud Drive as Documents/flockfinder_backup.sqlite. User preferences live in iCloud Key-Value Store — an invisible synced preferences file automatically managed by iOS.
