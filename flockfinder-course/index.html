The Secret Life of a Detection

What FlockFinder actually does — and the journey your data takes in the half-second after a camera is found.

What Is This App, Really?

Imagine you're driving around your neighborhood with a small electronic box plugged into your car. That box is constantly sniffing the air for Wi-Fi signals, looking for patterns that match known surveillance cameras — particularly Flock Safety cameras, which are ALPR cameras used to photograph license plates.

When that box finds one, it beams a message to your iPhone over Bluetooth. Your phone lights up, plays an alert, records the GPS coordinates, and saves everything to a private database that only you can see.

📡

The Hardware

A ESP32-S3 FeatherS3 device scans Wi-Fi signals and broadcasts detections over BLE.

📱

The App

This iOS app is the companion that receives detections, logs GPS location, and stores everything locally.

🗺️

The Map

Every detection appears as a point on your personal map — building a local picture of surveillance infrastructure.

The Orchestration Code

Every time the BLE hardware sends a detection, this function runs. It's the conductor — reading from one source and directing all the others.

CODE 

private func handleDetection(_ bleData: BLEManager.DetectionData) {
    let locationData = locationManager.getLocationData()
    let detection = FlockDetection(bleData: bleData, location: locationData)
    guard detection.confidence >= appSettings.minimumConfidence else { return }
    databaseManager.insertDetection(detection)
    if appSettings.audibleAlertsEnabled {
        audioManager.playDetectionAlert()
    }
}
PLAIN ENGLISH

When a detection arrives from Bluetooth, run this function...

Ask the GPS system for the current location right now.

Combine the camera data with the GPS data into one complete record.

If the detection's confidence score is too low, ignore it and stop here.

Save the full record to the database.

If the user has sound alerts turned on...

...play the detection sound.

Close the if-block.

End of function.

💡
The Coordinator Pattern

Notice how handleDetection doesn't do any of the work itself — it just calls other managers. This is a famous software pattern called the Coordinator pattern. When you ask an AI to "add a feature when a detection happens," this is the exact function to tell it to modify.

What Gets Saved for Every Detection

A single detection is far more than just "camera found here." It's a rich snapshot of your environment at that exact moment.

📸
Camera Identity

Device type (Flock Safety, Ring, Nest…), MAC address, and Wi-Fi SSID.

📍
Location

Latitude, longitude, altitude, and GPS accuracy — precise enough to place a map pin within a few meters.

🚗
Motion Context

Speed in mph, compass heading, and activity type (walking, driving, cycling) from the motion sensor.

📊
Signal Strength & Confidence

RSSI in dBm and a confidence score (0–1) estimating how certain the detection is.

Meet the Cast

Five specialist managers and one coordinator — each with a single job, each talking to the others through a careful chain of command.

The Team Behind Every Detection

Think of FlockFinder's app like a submarine crew — every person has one role, no one does someone else's job, and the captain coordinates them all without physically pulling the levers. If one person is asleep, the whole mission fails.

In code, each crew member is a class called a Manager. Here's the full roster:

📶

BLEManager

The Radio Operator. Speaks Bluetooth to the ESP32 hardware. Scans, connects, and receives detection messages.

📍

LocationManager

The Navigator. Tracks your GPS position, speed, compass heading, and updates whenever you move 5+ meters.

💾

DatabaseManager

The Archivist. Writes every detection to the on-device SQLite database, fetches history, handles exports.

🏃

MotionManager

The Speedometer. Uses the iPhone's accelerometer to detect if you're walking, driving, or stationary.

🔔

AudioAlertManager

The Alarm. Plays detection sounds and manages the haptic buzz — completely separate from the rest of the logic.

🎯

DetectionCoordinator

The Captain. Knows nobody else's job — just the order things need to happen. Calls all managers in sequence on every detection.

How the UI Stays in Sync

Each manager is declared as an ObservableObject. Their properties are marked @Published. This is the mechanism that makes your screen update the moment a new detection arrives.

CODE 

class BLEManager: NSObject, ObservableObject {
    @Published var isScanning = false
    @Published var isConnected = false
    @Published var connectionState: ConnectionState = .disconnected
    @Published var recentDetections: [FlockDetection] = []
    @Published var statusMessage: String = "Ready to scan"
}
PLAIN ENGLISH

Create a class called BLEManager. The "ObservableObject" part means views can watch it for changes.

Watchable: whether we're currently scanning for devices.

Watchable: whether we're connected to the ESP32.

Watchable: a description of the current state (disconnected, scanning, connected, etc).

Watchable: the list of recent detections — when this changes, the map and history update instantly.

Watchable: the status text shown to the user ("Ready to scan", "Found device"...).

End of class definition.

💡
The Bulletin Board Metaphor

Think of @Published properties as items pinned to a bulletin board. Any view that "subscribes" to the board automatically sees updates the moment something changes — without anyone manually notifying them. Engineers call this the Observer pattern.

03

Speaking Bluetooth

How the ESP32 hardware "talks" to your iPhone — the invisible protocol that delivers every detection across the air.

Bluetooth, but Make It Lean

Classic Bluetooth (in speakers and headphones) streams audio continuously — it's like a running water tap. Bluetooth Low Energy (BLE) is more like a postal service — small, precisely addressed packets, only when there's something to say.

This is perfect for FlockFinder: the hardware only needs to send a short JSON message when it detects a camera. No need for a constant stream.

1
Advertise

The ESP32 broadcasts a BLE advertisement — like holding up a sign that says "FlockFinder-S3 is here." Your iPhone sees it while scanning.

2
Connect

Your iPhone establishes a private BLE connection. It goes from "I see you" to "let's talk." Only one device can be connected at a time.

3
Discover Services

The iPhone asks: "What can you do?" The ESP32 lists its services. The FlockFinder service contains the detection channel.

4
Subscribe to Notifications

The iPhone subscribes to the detection characteristic: "Tell me every time you find something." From now on, the ESP32 pushes detections without being asked.

Channel Numbers: UUIDs

In BLE, every service and characteristic gets a unique ID called a UUID. Think of it like a radio channel number. Both devices must agree on the exact same numbers — if they don't match, they can't communicate.

These UUIDs are hardcoded in the iOS app and must match what's programmed into the ESP32 firmware:

4fafc201-…914b Service UUID — the "FlockFinder station" on the BLE radio. Both devices tune to this.
beb5483e-…26a8 Detection Characteristic — the specific channel that pushes camera detections as notifications (read-only, notify).
beb5483e-…26a9 Command Characteristic — a writable channel used to send commands back to the ESP32 (e.g., trigger a scan).
beb5483e-…26aa Stream Characteristic — a live feed of everything the ESP32 sees, used for the Debug Stream view.
CODE 

static let flockServiceUUID =
    CBUUID(string: "4fafc201-1fb5-459e-8fcc-c5c9c331914b")
static let detectionCharacteristicUUID =
    CBUUID(string: "beb5483e-36e1-4688-b7f5-ea07361b26a8")
static let commandCharacteristicUUID =
    CBUUID(string: "beb5483e-36e1-4688-b7f5-ea07361b26a9")
PLAIN ENGLISH

Create a permanent (static) UUID constant for the main FlockFinder service...

...using this exact 128-character ID. If the ESP32 firmware uses a different one, they won't connect.

Create a UUID for the detection channel (where camera alerts come from)...

...this ends in "a8" — it's the read/notify characteristic.

Create a UUID for the command channel (where we send instructions to the device)...

...this ends in "a9" — the next sequential ID, a writable characteristic.

The Message Format: JSON

When the ESP32 detects a camera, it sends a tiny text message in JSON format over the detection characteristic. The iOS app parses this JSON into a Swift object.

Here's what a real detection message looks like on the wire:

JSON from ESP32 

{
  "type": "Flock Safety",
  "mac": "AA:BB:CC:DD:EE:FF",
  "ssid": "FLOCK-CAM-001",
  "rssi": -55,
  "confidence": 0.95,
  "ts": 123456789
}
PLAIN ENGLISH

Start of the detection data package.

What kind of device: a Flock Safety ALPR camera.

Hardware ID of the camera (unique to this physical device).

The Wi-Fi network name the camera broadcasts — "FLOCK-CAM-001".

Signal strength: -55 dBm (strong — the closer to 0, the stronger).

How sure the ESP32 is that this is a Flock camera: 95% confident.

Timestamp from the ESP32 (seconds since it booted).

End of the package.

ℹ️
Why Text? Why Not Binary?

JSON is human-readable text, which makes debugging easy — you can literally read the message. The payload is tiny (under 200 bytes) so the minor overhead compared to packed binary doesn't matter. For FlockFinder's use case, readability wins over micro-optimization.

The Filing System

How FlockFinder stores thousands of detections permanently — without needing a server, an account, or a Wi-Fi connection.

The Library in Your Pocket

Think of a traditional library: books are organized by category, every book has an entry in the card catalog, and you can ask a librarian to find all books by a certain author, published before a certain year, in under a minute.

SQLite is that library — except it lives in a single file on your iPhone. There's no server, no internet connection needed, no account required. FlockFinder creates a file called flockfinder.sqlite in your phone's private storage the first time it runs.

📂

It's Just a File

The entire database lives in one .sqlite file on device. Back it up by copying the file. Delete the file and all data is gone.

🔍

You Query It with SQL

SQL lets you ask questions like: "Give me all Flock Safety detections from this week, ordered by time."

⚡

Built Into iOS

Apple ships SQLite3 with every iPhone. FlockFinder uses C-level SQLite3 bindings directly — no third-party library, no extra download.

How Detections Are Shaped in the Database

Every piece of data lives in a table — like a spreadsheet. Each column in the table stores one piece of information. Here's the exact schema FlockFinder creates:

SQL SCHEMA 

CREATE TABLE IF NOT EXISTS detections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    device_type TEXT NOT NULL,
    mac_address TEXT,
    rssi INTEGER NOT NULL,
    confidence REAL NOT NULL,
    latitude REAL NOT NULL,
    longitude REAL NOT NULL,
    speed REAL,
    timestamp TEXT NOT NULL,
    synced INTEGER DEFAULT 0
);
PLAIN ENGLISH

Create the "detections" table if it doesn't already exist (safe to run on every app launch).

Auto-incrementing unique number for each detection — ID 1, 2, 3, etc. Never reused.

The camera brand name — required, can't be empty. "Flock Safety", "Ring", etc.

Hardware MAC address — optional (may not always be available).

Signal strength — required integer (like -55 or -80).

Confidence score — required decimal number (like 0.95).

GPS latitude — required decimal (like 37.7749).

GPS longitude — required decimal (like -122.4194).

Speed in m/s — optional, in case GPS isn't tracking motion.

When this happened, as an ISO-8601 date string — required.

Has this been backed up to iCloud? 0 = no, 1 = yes. Defaults to 0.

End of table definition.

Speed Tricks: Database Indexes

Imagine searching all 2,000 books in a library by hand every time you want to find one. Or — using the card catalog, which is sorted alphabetically and tells you exactly which shelf to go to.

A database index is that card catalog. FlockFinder creates four of them:

idx_timestamp Makes "show me detections from the last week" instant instead of scanning every row.
idx_device_type Makes "filter by Flock Safety only" fast — used on the map and history views.
idx_synced Makes "find all unsynced detections for iCloud backup" fast without touching already-synced rows.
idx_location Makes map viewport queries fast — "find all cameras within this bounding box" uses lat/lng together.
💡
When to Ask AI to Add an Index

If your app ever shows a list that gets progressively slower as data grows, ask AI: "add a database index on [the column I filter by]." You'll hear engineers call slow unindexed queries a full table scan — a classic performance gotcha in mobile apps.

Prepared Statements: The Security Guard

Permission Granted

Why iOS makes your app beg for access — how the permission system works, and what happens when it says no.

iOS Has a Bouncer at Every Door

Imagine every sensitive hardware feature on your iPhone — GPS, microphone, camera, Bluetooth, motion sensors — has its own bouncer at the door. No app gets in without the user explicitly approving it. Even after approval, the user can revoke access at any time from Settings.

FlockFinder needs three types of hardware access to work. Each one requires the user's explicit consent:

📶

Bluetooth

Required to scan for and connect to the ESP32 hardware. Without this, the app literally cannot receive any detections.

📍

Location

Required to tag detections with GPS coordinates. The app requests "When In Use" — only while the app is open.

🏃

Motion & Activity

Optional but useful — lets the app record whether you were driving, walking, or stationary when each camera was detected.

Telling Apple Why You Need Access

iOS requires every app to explain why it needs each permission. These explanations live in a configuration file called Info.plist. Apple reviews them — and if they seem suspicious or vague, your app gets rejected from the App Store.

NSBluetoothAlwaysUsageDescription "FlockFinder needs Bluetooth to connect to your FlockFinder device." — Required for any BLE use on newer iOS.
NSLocationWhenInUseUsageDescription "FlockFinder needs your location to tag detected surveillance devices." — Shown when requesting foreground location.
NSLocationAlwaysAndWhenInUseUsageDescription For the optional background mode — "FlockFinder needs location access to log detections while you drive." Background requires extra justification.
NSMotionUsageDescription "FlockFinder uses motion to record your direction and speed during detection." — For CoreMotion access.
⚠️
The Missing Plist Key Crash

If an app calls a permission API without the matching Info.plist key, iOS immediately crashes the app with a hard error. BLEManager.swift even has a debug check: if Bundle.main.object(forInfoDictionaryKey: "NSLocationWhenInUseUsageDescription") == nil — it prints a warning before crashing so you know exactly what's missing. If you see cryptic permission crashes, Info.plist is always the first place to check.
